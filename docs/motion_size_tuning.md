# Stage1(MOG2)の動体サイズ範囲チューニング手順

## 設定場所

`configs/targets/<target>.yaml` の2つのキー:

```yaml
motion_min_area: 100     # これ未満の面積の動体はフラグしない（小さすぎる=ノイズとみなす）
motion_max_area: 25000   # これを超える面積の動体はフラグしない（照度変化等の巨大ブロブ除外）
```

`configs/pipeline.yaml`（全対象共通）ではなく`targets/*.yaml`（対象別）にあるのが正しい。
CLAUDE.mdで「対象ごとに変えてよい3点」の1つに指定されている項目（動体サイズ範囲）のため。
**「動画ごと」のチューニングはNGだが、「対象・撮影シナリオごと」のチューニングは想定内。**

## 単位に関する注意（重要）

この面積は**元動画の解像度ではなく、Stage1が内部でリサイズする幅（既定640px）でのピクセル²**。
リサイズ幅は`stage1_mog2_flag.py --resize-width`（既定640、現状config化していないCLI引数）。
これを変えたらmin_area/max_areaも合わせて再計算が必要（面積はリサイズ倍率の2乗で変わる）。

## 値の決め方（キャリブレーション手順）

1. 対象動画から、狙いたい最小サイズの訪花者が写っているフレームを1枚選ぶ
   （例: 今回の「小さい訪花者」なら、それが写っている代表フレーム）。
2. そのフレームを640px幅にリサイズする（`--resize-width`の既定値に合わせる）。
   ```python
   import cv2
   img = cv2.imread("frame.jpg")
   scale = 640 / img.shape[1]
   resized = cv2.resize(img, (640, int(img.shape[0]*scale)))
   cv2.imwrite("resized_640.jpg", resized)
   ```
3. リサイズ後の画像で、対象昆虫の外接矩形（バウンディングボックス）のおおよその幅・高さを
   ピクセル単位で読み取る（画像ビューアで座標を確認するか、適当な画像編集ツールで矩形を引く）。
4. `motion_min_area`の目安 = 幅×高さ×0.5〜0.7程度（MOG2の輪郭面積はバウンディングボックス
   面積より小さくなりがちなため、bbox面積そのままだと厳しすぎることが多い）。
   この値より少し余裕を持たせて小さめに設定する（対象個体のサイズにはばらつきがあるため）。
5. `motion_max_area`は現状デフォルト25000のままで基本問題ない
   （変更が必要になるのは、画面全体がフラグされる等の逆方向の異常が出た場合のみ）。

## 変更時の副作用（トレードオフ）

- `motion_min_area`を下げる → 小さい動体も拾えるようになる一方、フラグされるフレーム数
  （Stage2でYOLOにかける対象フレーム数）が増える → 処理時間が伸びる、ノイズ候補も増えうる。
- 極端に下げすぎると、葉の揺れや光の反射等の小さいノイズも拾ってしまい、Stage2/3の
  ノイズ候補が増える（Stage3のCLIPスコアリングで多くは弾かれる想定だが、レビュー量は増える）。

## 反映方法

1. `configs/targets/<target>.yaml`の`motion_min_area`を書き換える
   （既存対象の値を直接変えるか、シナリオ別に新しいtargetファイルを作るかは用途次第。
   例えば「同じbombusでも小型訪花者を拾いたい撮影地専用」の設定にしたい場合は、
   `bombus.yaml`をコピーして`configs/targets/bombus_small.yaml`のような別targetを作り、
   `--target bombus_small`で実行する方が、既存のbombus運用に影響を与えず安全）。
2. Stage1から再実行する（`stage1_mog2_flag.py`の出力はresumeでスキップされるため、
   既存の`stage1.json`があるディレクトリに書き込む場合は`--out-json`の出力先を変えるか、
   古い出力を消してから再実行する）。
